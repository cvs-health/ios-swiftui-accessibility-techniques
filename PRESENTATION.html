AI, Automation, and Shifting iOS Accessibility Left

Finding and fixing accessibility defects before testers do

The Problem: Found Too Late

  • A11y defects found by testers at end of SDLC are expensive to fix — context is lost, rework costs high, releases delayed
  • Developers don't get feedback until QA, sometimes weeks later
  • A missing .accessibilityLabel takes 30 seconds to fix while coding…
  • …but becomes a JIRA ticket, a sprint item, and a code review cycle when found in QA

Goal: give developers the same accessibility checks while they code

What is a11y-check?

  • Static analysis tool for SwiftUI — like axe-core for HTML
  • 31 rules mapped to 17 WCAG 2.2 success criteria
  • 4 impact levels: Critical, Serious, Moderate, Minor
  • WCAG scoring: 0–100 with letter grade (A+ through F)
  • Auto-fix for common issues (--fix)
  • Open source, built by CVS Health

Without AI: CLI Directly

No tokens, no API costs, runs offline

# Install
brew install a11y-check

# Scan all Swift files
a11y-check .

# Show only errors
a11y-check . --only error

# Scan a single file
a11y-check MyView.swift

# Auto-fix issues
a11y-check . --fix

# Preview fixes before applying
a11y-check . --fix --dry-run

# Generate HTML report
a11y-check . --format html > report.html

# Watch mode — re-runs on every file save
a11y-check . --watch

# See all 31 rules
a11y-check --list-rules

All features — scanning, auto-fix, scoring, HTML reports, Xcode & CI integration — work without AI
The AI integrations are optional enhancements

Full loop: detect → score → understand → fix → re-score

In Xcode: Every Build

  • Run Script build phase — add a11y-check as a build step in your Xcode project; works with any project type
  • SPM Build Plugin — add as a Swift Package Manager dependency; runs automatically with zero config
  • Issues appear inline in the editor, just like compiler warnings

CI/CD: Pull Requests

  • a11y-check GitHub Actions workflow: builds tool, runs scan, posts score as PR comment, fails if below threshold
  • SARIF output → GitHub Code Scanning: issues appear as annotations directly on PR diffs

Adopting on existing codebases

# Only flag NEW issues, not legacy debt
a11y-check Sources/ --diff --diff-base main --only error

# Save baseline, only fail on regressions
a11y-check Sources/ --baseline

HTML report uploaded as artifact for compliance review

Key Takeaway

Every developer touchpoint — editor, build, PR — has accessibility checking built in.

The same 31 rules run at every stage.

Defects are found and fixed minutes after they're written,
not weeks later by a tester.

AI is optional — the core tool works without tokens or API costs.
